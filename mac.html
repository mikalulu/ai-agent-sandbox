macOS / Lima Setup Guide

アーキテクチャ

⚠
Display 隔離について: XQuartz はホストの X11 サーバーを共有するため、VM 内エージェントがホスト画面へアクセスできる可能性がある。Linux 版の Xephyr による完全分離とは異なる。コンテナ起動中はホスト操作を避ける運用で補う。

02

Lima VM 定義ファイル

Lima は YAML で VM を定義する。ホスト FS のマウント範囲・CPU・メモリをここで制御する。

yaml — ~/.lima/agent/lima.yaml
# Lima VM 定義 — AI Agent Sandbox
vmType: vz              # Apple Virtualization.framework (M2 最速)
rosetta:
  enabled: false        # aarch64 ネイティブ実行

images:
  - location: "https://cloud-images.ubuntu.com/releases/24.04/release/ubuntu-24.04-server-cloudimg-arm64.img"
    arch: aarch64

cpus: 4
memory: "8GiB"
disk: "60GiB"

# ── ホスト FS マウント設定 ──────────────────────
mounts:
  # ホームは read-only (デフォルト挙動を明示)
  - location: "~"
    writable: false
  # 作業ディレクトリのみ read-write で公開
  - location: "~/lima-agent-work"
    writable: true

# ── ネットワーク ────────────────────────────────
# デフォルト: NAT (ホストネットワークには非公開)

# ── X11 forwarding ──────────────────────────────
# SSH -X で XQuartz に転送 (起動コマンドで指定)

containerd:
  system: false         # 今回は不使用
  user: false

作業ディレクトリ作成 (ホスト側)

bash
mkdir -p ~/lima-agent-work
03

VM 起動 / セットアップ

VM 作成 · 起動

bash
# lima.yaml を配置して VM 作成
mkdir -p ~/.lima/agent
cp lima.yaml ~/.lima/agent/

# VM 起動 (初回は Ubuntu イメージのダウンロードあり)
limactl start agent

# VM に入る (通常 SSH)
limactl shell agent

VM 内 — 基本セットアップ

bash — VM 内
sudo apt-get update && sudo apt-get install -y --no-install-recommends \
  # X11 / WM
  xauth x11-apps openbox dbus-x11 xdg-utils \
  # Electron / Chromium 共通ライブラリ
  libgbm1 libgl1-mesa-dri \
  libnss3 libatk1.0-0 libatk-bridge2.0-0 \
  libgdk-pixbuf2.0-0 libgtk-3-0 \
  libasound2 libxss1 libxtst6 \
  # フォント
  fonts-noto-cjk fonts-noto-color-emoji \
  # ユーティリティ / ビルドツール
  ca-certificates curl wget gnupg \
  build-essential git \
  # Python
  python3 python3-pip python3-venv

VM 内 — Node.js (nvm)

bash — VM 内
export NVM_DIR="$HOME/.nvm"
curl -fsSL https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source "$NVM_DIR/nvm.sh"
nvm install 22 && nvm alias default 22

VM 内 — Rust

bash — VM 内
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

Windsurf (.deb · arm64)

bash — VM 内
# arm64 版 .deb を指定 (M2 = aarch64)
curl -fsSL https://windsurf-stable.codeiumdata.com/linux-arm64/windsurf_latest_arm64.deb \
  -o /tmp/windsurf.deb
sudo dpkg -i /tmp/windsurf.deb || sudo apt-get install -fy
rm /tmp/windsurf.deb

Antigravity (.deb · arm64)

bash — VM 内
# URL は公開後に差し替え
curl -fsSL https://antigravity.google/download/antigravity_latest_arm64.deb \
  -o /tmp/antigravity.deb
sudo dpkg -i /tmp/antigravity.deb || sudo apt-get install -fy
rm /tmp/antigravity.deb
⚠
arm64 版の存在確認が必要。 Windsurf / Antigravity が arm64 Linux .deb を提供していない場合は、x86_64 版を Rosetta 2 経由で動かすか、CLI エージェントのみ使用する。uname -m で VM アーキテクチャを確認すること。
05

GUI 起動

Lima VM への SSH X11 forwarding で XQuartz に出力する。

XQuartz の事前設定 (ホスト側・初回のみ)

bash — ホスト
# XQuartz を起動し、設定 > セキュリティ で
# "ネットワーク・クライアントからの接続を許可" をオンにする
open -a XQuartz

# ホスト側 DISPLAY 確認
echo $DISPLAY   # → :0 または /tmp/launchd-*/com.apple.launchd.*/org.xquartz...

SSH X11 forwarding で VM に入り GUI 起動

bash — ホスト
# -X: X11 forwarding 有効
limactl shell --shell=/bin/bash agent -- bash -c '
  eval $(dbus-launch --sh-syntax)
  export DBUS_SESSION_BUS_ADDRESS
  openbox &
  sleep 0.5
  exec windsurf    # または antigravity
'
💡
Lima の limactl shell は内部的に SSH -X を使用するため、XQuartz が起動していれば DISPLAY が自動的に転送される。

CLI エージェントのみの場合 (X11 不要)

bash — ホスト
# GUI 不要なら通常 shell で OK
limactl shell agent
cd ~/lima-agent-work

# Ollama はホスト側で動いているので直接呼べる
export OLLAMA_HOST=host.lima.internal:11434
claude   # または codex / gemini / opencode / aider
06

Ollama — ホスト側で Metal 動作

✓
Ollama は macOS ホスト側で直接動作させる。Metal バックエンドで M2 GPU を使えるため VM 経由より高速。VM 内エージェントは host.lima.internal:11434 経由でアクセスできる。

Ollama サーバー起動 (ホスト)

bash — ホスト
# VM からのアクセスを受け入れるため 0.0.0.0 で Listen
OLLAMA_HOST=0.0.0.0:11434 ollama serve &

# モデル取得
ollama pull qwen2.5-coder:3b   # ~2GB · 軽量
ollama pull qwen2.5-coder:7b   # ~5GB · バランス

VM 内エージェントから Ollama を使う

bash — VM 内
# Lima が提供するホスト到達用の特殊 hostname
export OLLAMA_HOST=host.lima.internal:11434

# aider から Ollama モデルを使う例
aider --model ollama/qwen2.5-coder:7b

モデルサイズ目安 (M2 Unified Memory)

text
qwen2.5-coder:3b    ~2 GB    RAM 8GB+    軽量・高速
qwen2.5-coder:7b    ~5 GB    RAM 16GB+   バランス
llama3.2:3b         ~2 GB    RAM 8GB+    汎用
codellama:7b        ~4 GB    RAM 16GB+   コーディング特化
⚠
M2 は Unified Memory のため GPU/CPU がメモリを共有する。Ollama モデル + Lima VM (8GiB) + macOS の合計が物理 RAM を超えないよう注意。16GB RAM なら 7B モデルまでが安全圏。
07

隔離まとめ

リソースVM からのアクセス備考
Host Filesystem (~/) READ ONLY Lima デフォルト · lima.yaml で制御
作業ディレクトリ (~/lima-agent-work) READ / WRITE lima.yaml で明示マウント
Host Display (XQuartz) SHARED SSH X11 forwarding · Linux 版より弱い
Host Keyboard / Clipboard SHARED XQuartz 共有に伴う制約
Host Network NAT only Lima デフォルト NAT
Ollama (Metal) API のみ host.lima.internal:11434 経由
Host Kernel ISOLATED Lima VM (QEMU/vz) による仮想化
🔁
Linux 版との主な差分: Display / Clipboard は XQuartz 共有のため隔離が弱い。ホスト FS はデフォルト read-only で守られる。Kernel は VM で完全分離(Linux 版 Podman より強い)。Ollama は Metal で高速動作する。
08

Tips

VM の停止 / 削除

bash — ホスト
limactl stop agent      # VM 停止 (ディスクは保持)
limactl start agent     # 再起動
limactl delete agent    # VM 削除 (ディスクも消える)

作業ディレクトリのバックアップ

bash — ホスト
# ~/lima-agent-work はホスト側に実体があるのでそのままコピー可
cp -r ~/lima-agent-work ~/lima-agent-work-backup-$(date +%Y%m%d)

VM スペック変更

bash — ホスト
# VM を停止してから lima.yaml を編集 → 再起動
limactl stop agent
vim ~/.lima/agent/lima.yaml   # cpus / memory を変更
limactl start agent

Ollama を launchd で常駐させる

bash — ホスト
# brew services で自動起動 (OLLAMA_HOST は launchd 環境変数で別途設定が必要)
brew services start ollama
ℹ
brew services start ollama はデフォルトで 127.0.0.1:11434 で起動する。VM からアクセスするには OLLAMA_HOST=0.0.0.0:11434 の環境変数設定が必要。launchd の plist を編集して EnvironmentVariables に追加する。
