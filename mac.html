📝 Limaちゃんのお作法（VM 定義ファイル）

Lima は YAML というファイルで「どのくらいパワーを使うか」「どのフォルダに触っていいか」をお約束します✨

yaml — ~/.lima/agent/lima.yaml
# Lima VM 定義 — AI Agent Sandbox
vmType: vz              # Apple Virtualization.framework (M2 最速)
rosetta:
  enabled: false        # aarch64 ネイティブ実行

images:
  - location: "https://cloud-images.ubuntu.com/releases/24.04/release/ubuntu-24.04-server-cloudimg-arm64.img"
    arch: aarch64

cpus: 4
memory: "8GiB"
disk: "60GiB"

# ── ホスト FS マウント設定 ──────────────────────
mounts:
  # ホームは read-only (デフォルト挙動を明示)
  - location: "~"
    writable: false
  # 作業ディレクトリのみ read-write で公開
  - location: "~/lima-agent-work"
    writable: true

# ── ネットワーク ────────────────────────────────
# デフォルト: NAT (ホストネットワークには非公開)

# ── X11 forwarding ──────────────────────────────
# SSH -X で XQuartz に転送 (起動コマンドで指定)

containerd:
  system: false         # 今回は不使用
  user: false

作業ディレクトリ作成 (ホスト側)

bash
mkdir -p ~/lima-agent-work
03

🚀 立ち上げて準備しよう！（起動とセットアップ）

VM 作成 · 起動

bash
# lima.yaml を配置して VM 作成
mkdir -p ~/.lima/agent
cp lima.yaml ~/.lima/agent/

# VM 起動 (初回は Ubuntu イメージのダウンロードあり)
limactl start agent

# VM に入る (通常 SSH)
limactl shell agent

VM 内 — 基本セットアップ

bash — VM 内
sudo apt-get update && sudo apt-get install -y --no-install-recommends \
  # X11 / WM
  xauth x11-apps openbox dbus-x11 xdg-utils \
  # Electron / Chromium 共通ライブラリ
  libgbm1 libgl1-mesa-dri \
  libnss3 libatk1.0-0 libatk-bridge2.0-0 \
  libgdk-pixbuf2.0-0 libgtk-3-0 \
  libasound2 libxss1 libxtst6 \
  # フォント / ロケール
  fonts-noto-cjk fonts-noto-color-emoji locales \
  # ユーティリティ / ビルドツール
  ca-certificates curl wget gnupg unzip jq yq ripgrep fd-find vim \
  # ブラウザ (Antigravity 用)
  chromium-browser \
  # デザイン / 図表作成
  graphviz inkscape \
  build-essential git \
  # Python
  python3 python3-pip python3-venv \
  && sudo sed -i 's/# ja_JP.UTF-8/ja_JP.UTF-8/' /etc/locale.gen \
  && sudo locale-gen
ℹ
日本語ロケールを有効にするには、~/.bashrc に export LANG=ja_JP.UTF-8 を追加してください。

VM 内 — Node.js (nvm)

bash — VM 内
export NVM_DIR="$HOME/.nvm"
curl -fsSL https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source "$NVM_DIR/nvm.sh"
nvm install 22 && nvm alias default 22

VM 内 — Rust

bash — VM 内
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

05

🖥️ 画面（GUI）を出してみよう！

Lima VMの中から「画面だけパソコン側に送って！」とお願い（X11 forwarding）して、XQuartzに映してもらいます📺✨

XQuartz の事前設定 (ホスト側・初回のみ)

zsh — ホスト
# XQuartz を起動し、設定 > セキュリティ で
# "ネットワーク・クライアントからの接続を許可" をオンにする
open -a XQuartz

# ホスト側 DISPLAY 確認
echo $DISPLAY   # → :0 または /tmp/launchd-*/com.apple.launchd.*/org.xquartz...

SSH X11 forwarding で VM に入り GUI 起動

zsh — ホスト
# -X: X11 forwarding 有効
limactl shell --shell=/bin/bash agent -- bash -c '
  eval $(dbus-launch --sh-syntax)
  export DBUS_SESSION_BUS_ADDRESS
  openbox &
  sleep 0.5
  exec windsurf    # または antigravity
'
💡
Lima の limactl shell は、実は裏でコッソリ画面転送のお願い（SSH -X）をしてくれています🤫 だからXQuartzさえ立ち上がっていれば、自動で画面がポンッと出てきますよ✨

CLI エージェントのみの場合 (X11 不要)

zsh — ホスト
# GUI 不要なら通常 shell で OK
limactl shell agent
cd ~/lima-agent-work

# Ollama はホスト側で動いているので直接呼べる
export OLLAMA_HOST=host.lima.internal:11434
claude   # または goose / gemini / opencode / aider
06

🤖 Ollama — Mac本来の力を使おう！

✓
Ollama は Mac本体側（ホスト側）で直接動かします💻 こうするとMacの強力なGPU（Metal）がフルパワーで使えるので、VMの中で動かすよりずっと速いんです🚀 中のAIたちからは host.lima.internal:11434 という魔法の宛先で呼び出せますよ♪

Ollama サーバー起動 (ホスト)

zsh — ホスト
# VM からのアクセスを受け入れるため 0.0.0.0 で Listen
OLLAMA_HOST=0.0.0.0:11434 ollama serve &

# モデル取得
ollama pull qwen2.5-coder:3b   # ~2GB · 軽量
ollama pull qwen2.5-coder:7b   # ~5GB · バランス

VM 内エージェントから Ollama を使う

bash — VM 内
# Lima が提供するホスト到達用の特殊 hostname
export OLLAMA_HOST=host.lima.internal:11434

# aider から Ollama モデルを使う例
aider --model ollama/qwen2.5-coder:7b

モデルサイズ目安 (M2 Unified Memory)

text
qwen2.5-coder:3b    ~2 GB    RAM 8GB+    軽量・高速
qwen2.5-coder:7b    ~5 GB    RAM 16GB+   バランス
llama3.2:3b         ~2 GB    RAM 8GB+    汎用
codellama:7b        ~4 GB    RAM 16GB+   コーディング特化
⚠
MacのメモリはGPUとCPUで一緒のタンク（Unified Memory）を使っています🚰 OllamaとLima VM、そしてMac本体の消費分を合わせて、タンクがからっぽにならないように気をつけてくださいね！ メモリが16GBなら、7Bサイズくらいのモデルまでが安心です✨

🛡️ ガードのまとめ (Isolation)

リソースVM からのアクセス備考
Host Filesystem (~/)READ ONLYLima デフォルト · lima.yaml で制御
作業ディレクトリ (~/lima-agent-work)READ / WRITElima.yaml で明示マウント
Host Display (XQuartz)SHAREDSSH X11 forwarding · Linux 版より弱い
Host Keyboard / ClipboardSHAREDXQuartz 共有に伴う制約
Host NetworkNAT onlyLima デフォルト NAT
Ollama (Metal)API のみhost.lima.internal:11434 経由
Host KernelISOLATEDLima VM (QEMU/vz) による仮想化
🔁
Linux版との主な違い🐧🍎: 画面やクリップボードのガードは少しゆるめです💦 でも、パソコンのファイルは基本「見るだけ（read-only）」でガッチリ守られ、システムの中枢（Kernel）はVMで完全に切り離されているのでとても安心です🛡️✨ Ollamaが高速で動くのもMacならではですね！

💡 便利なヒント集 (Tips)

VM の停止 / 削除

zsh — ホスト
limactl stop agent      # VM 停止 (ディスクは保持)
limactl start agent     # 再起動
limactl delete agent    # VM 削除 (ディスクも消える)

作業ディレクトリのバックアップ

zsh — ホスト
# ~/lima-agent-work はホスト側に実体があるのでそのままコピー可
cp -r ~/lima-agent-work ~/lima-agent-work-backup-$(date +%Y%m%d)

VM スペック変更

zsh — ホスト
# VM を停止してから lima.yaml を編集 → 再起動
limactl stop agent
vim ~/.lima/agent/lima.yaml   # cpus / memory を変更
limactl start agent

Ollama を launchd で常駐させる

zsh — ホスト
# brew services で自動起動 (OLLAMA_HOST は launchd 環境変数で別途設定が必要)
brew services start ollama
ℹ
普通に brew services start ollama をすると、Mac本体からしか話しかけられないお留守番モード（127.0.0.1）になってしまいます🏠 中のVMからも話しかけられるようにするには、設定（plist）を少し書き換えてあげる必要がありますよ✨ (EnvironmentVariables に OLLAMA_HOST=0.0.0.0:11434 を追加します)

🔍 トラブルシューティング

XQuartz が "connection refused" を返す / DISPLAY が空

XQuartz が起動していないか、"Allow connections from network clients" が無効です。

zsh — ホスト
# XQuartz を起動
open -a XQuartz
# Network client 接続を許可（要ログアウト→再ログイン）
defaults write org.xquartz.X11 nolisten_tcp -bool false
# DISPLAY を確認
echo $DISPLAY   # :0 などが表示されれば OK
limactl start agent が失敗する / タイムアウトする

デバッグモードで起動してエラーログを確認します。

zsh — ホスト
limactl start --debug agent 2>&1 | tail -80

VM ディスク容量が不足している場合は lima.yaml の disk を増やして再作成してください。

zsh — ホスト
limactl delete agent
limactl start agent
VM 内から Ollama (localhost:11434) に繋がらない

Ollama はデフォルトで 127.0.0.1 にしかバインドしないため、VM 内から到達できません。

zsh — ホスト
# brew services を止めて、0.0.0.0 バインドで再起動
brew services stop ollama
OLLAMA_HOST=0.0.0.0:11434 ollama serve &
arm64 / aarch64 エラー — x86_64 バイナリを実行しようとしている

lima.yaml の arch: aarch64 が正しいか確認してください。インストールスクリプトが x86_64 バイナリを DL している場合は arm64 版 URL を使ってください。

bash — Lima VM
uname -m   # aarch64 と表示されれば OK
ollama pull が途中で止まる

Ctrl-C で中断して再実行すると続きからダウンロードされます。

zsh — ホスト
ollama pull qwen2.5-coder:7b  # 再実行で続きから再開
