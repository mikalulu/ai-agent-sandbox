AI Agent 動作環境マトリクス

Shell とファイルの互換性を OS ごとに整理した早見表です。
🌸 OS別のおすすめセッティング（ベストプラクティス）
🐧
Linux
直接インストール / VM — 最も互換性が高い環境です
推奨構成
bash または zsh  ×  ext4 / xfs / btrfs
AI の学習データの多くは Linux/bash 環境のものです。そのため AI の出力と実環境の整合性が最も高く、パスや権限も AI の予測通りに動作します。
🐚 Shell 選択
bash を推奨します。デフォルトシェルを /bin/bash にしておけば、AI が生成する bash 構文がそのまま動作します。
📁 作業ディレクトリ
~/（ホームディレクトリ）配下を推奨します。/tmp は再起動で消去されるため長期作業には不向きです。ホームディレクトリ内であれば権限エラーも発生しにくくなります。
💾 FS 選択
どのファイルシステムでも AI からの見え方に大きな差はありません。ただし btrfs のスナップショット等を使用すると実パスにずれが生じる場合があるため注意してください。
⚠️ 罠: sudo 環境でのパス差異
sudo を使うと PATH が変わり、ツールが見つからなくなることがあります。sudo -E を使うか、visudo で環境変数を引き継ぐ設定をしてください。
⚠️ 罠: コンテナ内の root 実行
Docker コンテナ内はデフォルトで root 権限です。AI が権限を変更すると、ホスト側のファイル権限にも影響する場合があるため注意が必要です。
🍎
macOS
ネイティブ環境には制約がありますが、Docker や Lima の併用で解消できます。
推奨構成
zsh / bash  ×  Lima または Docker
macOS ネイティブでは case-insensitive FS や SIP による制約があります。Lima VM を使うことでこれらを回避でき、macOS では最も推奨される構成です。
🐚 Shell 選択
標準の zsh で問題ありません。プリインストールの /bin/bash はバージョンが古く、AI が生成するスクリプトが動作しない場合があります。Homebrew で新しい bash を導入するか、zsh を使用してください。
🔮 Lima の優位性
Lima は軽量な Linux VM です。直接 VM 内で作業することで、Docker の DinD 制約を受けずに Linux 環境を利用できます。
⚠️ 罠: APFS case-insensitive
APFS は File.txt と file.txt を同一ファイルとして扱います。AI が Linux の前提で操作すると Git で不整合が生じるため、core.ignorecase の設定に注意してください。
⚠️ 罠: SIP による無音失敗
SIP（System Integrity Protection）により保護されたパスは書き換え不可です。AI が変更を試みてもエラーが出ずにサイレントに失敗するため、問題の特定が困難になります。
⚠️ 罠: Firmlink によるパスずれ
一部のパス（/var、/tmp 等）は Firmlink により別の場所にリダイレクトされています。AI が実パスを解決した際にずれが生じることがあります。
⚠️ 罠: Docker ボリュームマウント
macOS のディレクトリをコンテナにマウントすると、case-insensitive の問題も持ち込まれます。コンテナ内に専用の作業ディレクトリを作成するのが安全です。
🪟
Windows
ネイティブ環境は互換性の問題が多いため、WSL2 の利用を推奨します。
推奨構成
WSL2 (bash / zsh)  ×  Linux FS (~/projects/)
Claude Code 等は Windows ネイティブ（コマンドプロンプト等）では正常に動作しないことが多いです。WSL2 を使えば、ほぼネイティブ Linux と同等の環境で実行できます。
🌊 WSL2 のセットアップ
wsl --install で Ubuntu を導入できます。Windows Terminal との組み合わせで使いやすくなります。
⚠️ 罠: 作業 FS を Windows 側に置く
⚠️ 重要: Windows 側のディレクトリ（C ドライブ等）で作業すると、9P プロトコル経由のため大幅に速度が低下します。必ず Linux 側（~/ 等）で作業してください。
⚠️ 罠: git の混在
Windows 側と WSL2 側の Git は別のインストールです。改行コードの自動変換を防ぐため、WSL2 側で設定してください。
⚠️ 罠: ロケール未設定
インストール直後はロケールが未設定の場合があります。文字化けを防ぐため、UTF-8 ロケールを設定してください。
🐚 git-bash の位置付け
Windows 向けの簡易 bash 環境ですが、パフォーマンスが低く、システム機能も不足しています。AI エージェントの実行環境としては不十分です。
🚫 CMD / PowerShell 5.1 は避ける
CMD は AI が生成する bash/POSIX 構文と互換性がありません。PowerShell を使用する場合は、新しい pwsh（バージョン 7 以上）を使用してください。

📎 Notes — Supplemental
SVN
基本的なコマンドは実行できますが、Git に比べて学習データが少ないため精度が低くなります。複雑な操作では誤りが多いため、SVN を使用する際は人間による確認が必要です。
ロケール設定
LANG が未設定だと日本語が文字化けし、AI が正しく処理できません。ファイルも UTF-8 に統一するのが最もトラブルの少ない構成です。
CP932 と CRLF — Windows のエンコーディング問題
CP932 の文字化け以上に厄介なのが、Windows 固有の CRLF 改行コードです。Git の差分が崩れる原因になるため、core.autocrlf=false に設定してください。
CI/CD 環境での注意点
CI/CD は通常 Linux + bash 環境のため、互換性は高いです。非対話実行（-y フラグ等）やシークレット管理など、自動実行向けの設定に注意してください。
