🐧
Linux
直接インストール / VM — 一番おすすめ！一切の制限なく快適に動きます✿
おすすめの構成✨
bash または zsh  ×  ext4 / xfs / btrfs
AIが学習しているデータの多くはLinuxやbashのものです🐧 だからAIの考えと実際の環境が一番ピッタリ合う最高のOSなんです✨ ファイルの場所や権限なども、AIの予測通りに完璧に動いてくれますよ！
🐚 Shell 選択
基本は bash を使いましょう✨ デフォルトシェルを /bin/bash にしておけば、AIが得意なbash特有の書き方もエラーにならずスルスル動きます♪
📁 作業ディレクトリ
~/ （ホームディレクトリ）の下に置きましょう🏠 /tmp は再起動で消えてしまうので長期間の作業には向きません💦 自分のホームの中なら権限のエラーも起きにくくて安心です！
💾 FS 選択
どれを選んでもAIからは同じように見えます👀 ただ、btrfsの特殊な機能（スナップショット等）を使うとファイルの本当の場所が少しズレて見えることがあるので、そこだけ気をつけてくださいね！
⚠️ 罠: sudo 環境でのパス差異
sudo を使うと、今まで見えていた便利なツールが突然見えなくなることがあります💦 sudo -E を使うか、設定（visudo）で環境を引き継げるようにしてあげましょう♪
⚠️ 罠: コンテナ内の root 実行
Dockerの中は基本的に一番えらい「root」になっています👑 AIが気まぐれに権限を変えたりすると、元のパソコン側のファイル権限まで変わってしまうことがあるので要注意です⚠️
🍎
macOS
そのまま使うと少し罠があります — DockerやLimaを使えば安心です♪
おすすめの構成✨
zsh / bash  ×  Lima または Docker
macOSそのまま（native）だと、大文字小文字の区別や強力なシステム保護（SIP）による制限があってAIが少し困ってしまいます💦 Lima というVMを使えば、それらの制限を回避できるのでmacOSでは一番おすすめです🍎✨
🐚 Shell 選択
標準の zsh のままで大丈夫です🙆‍♀️ 最初から入っている /bin/bash は古すぎてAIのやりたいことが上手く動かないことがあります💦 Homebrewで新しいbashを入れるか、zshで進めましょう！
🔮 Lima の優位性
Lima はとっても軽快なLinuxの仮想マシンです✨ 直接ここに入ることで、Docker特有のややこしい制限（DinD）を受けずに、のびのびとLinux環境の恩恵を受けられますよ♪
⚠️ 罠: APFS case-insensitive
File.txt と file.txt を同じファイルだと勘違いしてしまいます💦 AIがLinuxの感覚で扱うとGitでおかしくなるので、早めに気づけるように設定しておきましょう！
⚠️ 罠: SIP による無音失敗
強固なシステム保護（SIP）により大事な場所は絶対に書き換えられません🛡️ AIが変更しようとしても、エラーが出ないままヌルッと失敗してしまうので少し厄介です💦
⚠️ 罠: Firmlink によるパスずれ
一部のフォルダ（/var や /tmp）は別の場所への「秘密のワープトンネル」になっています🌀 AIが本当の場所を調べたときにズレが生じて混乱することがあります👀
⚠️ 罠: Docker ボリュームマウント
macOSのフォルダをそのままコンテナに繋ぐと、大文字小文字の勘違い問題まで一緒に持ち込まれてしまいます💦 コンテナの中に専用の作業場所を作るのが一番安全です✨
🪟
Windows
そのままでは難しいです — WSL2 を使うのが一番の解決策です♪
おすすめの構成✨
WSL2 (bash / zsh)  ×  Linux FS (~/projects/)
実は Claude Code などは Windowsのまま（コマンドプロンプト等）では正しく動かないことが多いです💦 WSL2を使ってLinux環境を呼び出せば、本物のLinuxとほぼ同じ快適さでサクサク動かせますよ🪟✨
🌊 WSL2 のセットアップ
wsl --install という魔法のコマンド一発でUbuntuが入ります🐧 Windows Terminal を使うと見た目もスッキリしてとっても使いやすくなりますよ！
⚠️ 罠: 作業 FS を Windows 側に置く
【一番大事なポイント⚠️】Windows側のフォルダ（Cドライブなど）で作業すると、通信の都合で恐ろしく動作が遅くなってしまいます🐢 必ずLinux側（~/ 等）で作業するようにしてくださいね！
⚠️ 罠: git の混在
Windows側のGitと、WSL2(Linux)側のGitは別々の存在です👀 余計な改行コードの変換をされないように、WSL2側でしっかり設定をしておきましょう♪
⚠️ 罠: ロケール未設定
入れた直後は「あなたが何語を話すか」が未設定なことがあります🗣️ 文字化けを防ぐために、しっかり UTF-8（日本語か英語）を設定しておいてくださいね✨
🐚 git-bash の位置付け
Windows向けに作られたお手軽なbash環境ですが、動きが少しモッサリしていて本格的なシステム機能が足りません🐢 AIのお家としては少し物足りない環境です💦
🚫 CMD / PowerShell 5.1 は避ける
昔ながらのコマンドプロンプト(CMD)はAIが得意な言葉をあまり理解してくれません🙅‍♀️ どうしてもPowerShellを使うなら、新しい pwsh（バージョン7以上）を使ってあげてくださいね🦋

📝 Notes — Important
1
Git の必須性（エージェントの感覚 / 記憶）
表からは省いていますが、AIエージェントにとって Git は生きていくために絶対に欠かせないインフラ です✨ 単なる履歴管理ではなく、「今どのファイルがどうなっているか」「自分がどこを変えたか」を理解し、失敗したら「安全に元に戻す」ための大切な短期記憶や感覚器官の役割を果たします🧠 Git（.gitフォルダ）がない場所だと、AIは目隠しされた状態になってしまい、実力をほとんど発揮できなくなってしまいます💦

Shell 構文非互換
AIはGitHubの膨大なLinux/bashのデータを食べて育ちました📚 だから息をするようにbash特有の書き方（配列やかっこ等）を使ってしまいます。これを他のシェル（cmdやfish等）にそのまま渡すと「なんのこと？」とエラーになって激突してしまいます💥

Claude Code: WindowsにWSLを入れてbash環境を作ってあげることで優しく回避できます✨
その他のエージェント: エラーを見て「あれ、今の間違ってたかも」と自分で直せる賢い子なら挽回できるかもしれません👀

コンテナの中からさらにDockerをいじる、少しマトリョーシカのようなやり方です🐳 便利ですがセキュリティの穴になりやすいのが玉にキズです💦 Limaを使えばこの複雑な構造を避けられるので、より高い評価になっていますよ✨

PowerShell 5.1 と pwsh の違い
昔から入っている青い画面（PowerShell 5.1）と、黒い画面の新しい子（PowerShell 7+ / pwsh）は全くの別物です！ 古い方を使うと、変な文字コードでファイルが出力されてAIが文字化けにうなされるので、必ず新しい pwsh を使ってあげてくださいね🦋

📎 Notes — Supplemental
SVN
コマンドを出すこと自体はできますが、なにせGitに比べて学習データが少なすぎるので、いまいち自信が持てないようです🐢 複雑なことはよく間違えるので、SVNを使う時は人間が手取り足取り確認してあげる必要があります💦
ロケール設定
言語設定（LANG）をきちんとしてあげないと、日本語が全てエイリアン語👽に文字化けしてしまい、AIがパニックになってしまいます💦 ファイル自体も UTF-8 という世界共通の文字コードに揃えておくのが一番トラブルが少なくて平和ですよ✨
CP932 と CRLF — Windows のエンコーディング問題
ターミナルの文字化け（CP932）より厄介なのが、Windows特有の「改行コード（CRLF）」のいたずらです👻 何もしていないのにGitの差分がぐちゃぐちゃになることがあるので、設定でしっかり自動変換をOFF（false）にしておきましょうね♪
CI/CD 環境での注意点
CI/CDの裏側はだいたい Linux + bash なので、環境としては最高に恵まれています✨ 途中で人間が「Yes/No」を入力しなくていいようにしたり（-yフラグ等）、秘密のパスワードが漏れないようにしたりと、無人で走らせるためのお作法だけ気をつけてくださいね🤖🚀
